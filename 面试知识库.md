## 1. "熟练使用 Python，具备面向对象设计与模块化开发能力"

### 这是什么意思

不是"会写 if/else/for"，而是能用 Python 的高级特性写出结构清晰、可扩展的代码。

### 我项目里的证据

| 特性 | 用法 | 在哪 |
|------|------|------|
| 类型注解 | `def chat(self, messages: list[ChatMessage | dict], tools: Optional[list]) -> ChatResponse:` | `model_client.py` |
| 装饰器 | `@timer` 自动统计耗时；`@retry` 指数退避重试 | `reflection/skill.py`、`model_client.py` |
| dataclass | `PipelineStep`、`OrchestratorResult` 全用 | `orchestrator.py` |

### 面试可能问

**Q：你项目里怎么用面向对象的？**

> A：我定义了 `BaseSkill` 抽象基类，13 个技能全部继承它，必须实现 `execute()` 方法。编排器调用时只依赖基类接口，不关心具体是哪个技能。这保证了新增技能零侵入，所有技能 100% 架构一致。

**Q：抽象基类和 Protocol 有什么区别？**

> A：抽象基类要求必须继承，Protocol 只看有没有某个方法。我在 ReflectionSkill 里用 Protocol 约束 LLM 客户端——不管是什么客户端，只要有 `.chat()` 就行，不需要强制继承我的类。

## 2. "策略模式解耦反馈逻辑"

### 通俗解释

不写 if-else 判断"现在是 RAG 还是代码审查"，而是建一本字典 `{"rag_answer": RAG检查函数, "code_explainer": 代码检查函数}`。ExternalFeedbackNode 查字典调用，不需要知道有多少种技能。

### 类比

就像遥控器——不是"如果是电视就调台、如果是空调就调温"写死，而是按了哪个键就执行哪个功能，加新电器不需要改遥控器。

### 面试可能问

**Q：为什么不用 if-else？**

> A：if-else 的问题是每加一种技能就要改 ExternalFeedbackNode 的代码，违反了开闭原则。字典映射让新增技能只加一行配置，不改原有代码。

**Q：你这算策略模式吗？**

> A：算。策略模式的核心是把"选择哪个算法"和"算法本身怎么执行"解耦。我的 FEEDBACK_STRATEGIES 字典就是"选择"，每个 `_rag_feedback` 函数就是"策略"。

## 3. "掌握 Java 集合框架、多线程、IO 流"

### 集合框架

不是"用过 ArrayList"，是知道什么场景选什么容器。

| 场景 | 用什么 | 原因 |
|------|--------|------|
| 展示用户列表 | `ArrayList` | 有序、按索引查快 |
| 判断用户名是否重复 | `HashSet` | O(1) 查，不需要关心顺序 |
| 缓存 id→User 对象 | `HashMap` | O(1) 查，key-value 天然匹配 |
| 多线程下共享的 Map | `ConcurrentHashMap` | 线程安全 |

面试说法：_"选容器看场景。有序用 List，去重用 Set，key-value 映射用 Map，多线程下用 ConcurrentHashMap。"_

### 多线程

**不是要你手写 new Thread，Spring Boot 已经在用了。**

Tomcat 默认 200 个线程池，每个 HTTP 请求分配一个线程。你的 Controller 代码天然跑在多线程环境里。

### IO 流

查数据库就是 IO——JDBC 从网卡读 MySQL 返回的二进制数据，MyBatis 帮你封装成对象。Spring 返回 JSON 也是 IO。

**你的项目里处处都有 IO 和多线程，只是被框架封装了，你没感知到。**

### 面试可能问

**Q：你的 Service 是线程安全的吗？**

> A：只要 Service 没有成员变量（共享状态），每个方法只用局部变量和参数，就是安全的。如果有共享数据（如计数器、缓存），需要用 synchronized、ConcurrentHashMap 或数据库行锁保护。

## 4. 线程安全详解

### 为什么"无状态就安全"

你写的方法只依赖参数和局部变量，不依赖任何"上次执行剩下来的东西"，每次调用完全独立。

就像两台自助取票机——各取各的票，互不干扰。

### 什么时候不安全

加了共享变量（成员变量 / 静态变量）。

```java
private int loginCount = 0;  // ← 成员变量，所有线程共享

public String login(String name) {
    loginCount++;  // ← 不是一步操作！读→加→写，三步之间可能被其他线程打断
}
```

两个请求同时来，读到的旧值相同 → 两个都加 1 → 都写回 11，而不是 12。

### 四种解决方案

| 方案 | 代码 | 什么时候用 |
|------|------|-----------|
| `synchronized` | `synchronized(lock) { ... }` | 有一段代码不能同时执行 |
| `AtomicInteger` | `new AtomicInteger(0).incrementAndGet()` | 简单自增计数 |
| `ConcurrentHashMap` | `new ConcurrentHashMap<>()` | 共享的 Map，读写都线程安全 |
| 数据库行锁 | `SELECT ... FOR UPDATE` | 抢同一行数据（库存扣减），交给 MySQL |

### 一句话记：

> **Controller 本身无状态所以安全。有共享变量就有并发问题。解决方式：synchronized 排队、AtomicInteger 原子操作、ConcurrentHashMap 安全容器、数据库行锁交给 MySQL。**

## 5. Spring Boot 项目的真实理解

### 你的理解（对的）

三层架构：Controller（接请求）→ Service（业务逻辑）→ Mapper（查数据库），加一个 Entity 实体层对应数据库表。Maven 管依赖，注解简化代码。

### 补充：隐式的多线程和 IO

| 你以为没接触 | 实际一直在用 |
|-------------|------------|
| 多线程 | 每个 HTTP 请求就是一个独立线程，Tomcat 线程池 200 个 |
| IO 流 | 查数据库是网络 IO，返回 JSON 是输出流，上传文件是输入流 |

### Spring 帮你屏蔽了什么

- 接收 HTTP 请求（Socket 监听）→ `@GetMapping` 替你做了
- 把请求体变成 Java 对象 → `@RequestBody` 替你做了
- 把 Java 对象变成 JSON 返回 → `@ResponseBody` 替你做了
- 并发请求的分发 → Tomcat 线程池替你做了
- 数据库连接管理 → 连接池替你做了

**所以你不需要写底层代码，但需要知道底层在干什么，以及有什么并发风险。**

## 6. LangGraph StateGraph 工作流编排

### 这是什么

不是手写 `A然后B然后C`，而是用图的方式定义节点（Node）和边（Edge）。LangGraph 自动按图执行。

### 我项目里的体现

```python
graph = StateGraph(GraphState)
graph.add_node("translate", TranslatorNode(...))
graph.add_node("summarize", SummarizerNode(...))
graph.add_edge("translate", "summarize")  # translate 完了自动走 summarize
graph.set_entry_point("translate")
graph.add_edge("summarize", END)
```

8 条流水线通过 PIPELINE_REGISTRY 配置化声明，调用时自动构建图。

### 条件边

```python
graph.add_conditional_edges(
    "critic",                          # 从 Critic 节点出发
    build_should_continue(config),     # 判断函数：返回 "reviser" 或 END
    {"reviser": "reviser", END: END},  # 根据返回值路由
)
```

效果：质量不达标 → 回到 Reviser 修订；达标 → 结束。

## 8. Reflection 反思模式

### 完整闭环

```
① GeneratorNode → 调用目标技能（如 rag_answer），产出初始答案
② ExternalFeedbackNode → 不调 LLM，纯程序检查：
   - RAG 场景：验证答案里引用了多少条检索文档
   - 代码场景：跑 flake8 检查原始代码
③ CriticNode → LLM（温度 0.1，极低温保证稳定）生成结构化批评：
   {severity: "major", description: "未引用文档3的关键数据", suggested_fix: "..."}
④ 条件边判断 → 质量 >= 0.8？收敛？迭代 >= 2次？
   任一满足 → END，否则 → ⑤
⑤ ReviserNode → LLM（温度 0.5）基于批评修订答案
   修订完 → 回到 ③ 让 Critic 再评
```

### 面试可能问

**Q：为什么要低温 0.1 的 Critic？**

> A：Critic 需要绝对客观稳定，找出所有问题，不能有随机性。高温会让它"今天觉得行、明天觉得不行"，没有参考价值。

**Q：怎么判定收敛？**

> A：比较连续两次迭代的 improvement_score。如果第二次改进幅度 < 0.05（基本没进步了），说明再改也没用，提前终止。


## 15. 数据库设计：10 个集合的关系

### 核心只有 3 个主角

```
用户（users）    帖子（posts）    评论（comments）
```

剩下 7 个全是"他们三个之间的互动记录"：

```
点赞（likes）       收藏（collects）    关注（follows）
消息（messages）    通知（notifications）  挑战（challenges）  排行（rankings）
```

### 三种关系记住就够

**① 一对多 — 用户 → 帖子**

```
posts 表：
  userId: "user_001"  ← 这个就叫"外键"，告诉你是谁发的
```

查"张三所有帖子"：`db.posts.where({ userId: "user_001" })`

**② 一对多 — 帖子 → 评论**

```
comments 表：
  postId: "post_001"  ← 这条评论挂在哪篇帖子下面
```

**③ 多对多 — 需要中间表**

"张三点赞了帖子 A、B" + "帖子 A 被张三、李四点赞" — 不能一对多。

```
likes 表（中间表）：
  userId: "user_001",  postId: "post_001"
  userId: "user_001",  postId: "post_002"
  userId: "user_002",  postId: "post_001"
```

双向都能查：谁点赞了什么 / 这篇被谁点赞了。

### 面试怎么讲

> "核心三张表：用户、帖子、评论。点赞、收藏、关注都是多对多关系，通过中间表关联。一对多用外键字段，多对多用独立的中间表。"

## 16. 推荐引擎完整版（策略模式 + 四规则 + 三大优化）

### 整体架构

```
引擎核心（RecommendationEngine）
  ├── 热度规则（HotRule）     ← 互动×时间衰减，权重 30%
  ├── 个性化规则（PersonalRule） ← 标签匹配度，权重 35%
  ├── 协同过滤规则（CFRule）   ← 浏览历史加权，权重 20%
  └── 运营加权规则（BoostRule） ← 人工干预，权重 15%
```

每条规则独立算分，加权求和 = 最终得分，按分排序。

### 规则 1：热度规则（时间衰减）

```
热度 = (赞×1 + 评论×2 + 收藏×1.5 + 分享×2.5) ÷ (1 + 发布天数)

例子：
  帖子A：50赞 10评论，发 1 天 → (50+20)÷2 = 35 分
  帖子B：100赞 20评论，发 7 天 → (100+40)÷8 = 17 分
  B 互动更多，但时间久了分数被压低
```

**为什么权重不同**：赞×1（点一下最简单）、评论×2（愿意打字是真感兴趣）、收藏×1.5（觉得有用）、分享×2.5（愿意分享最有价值）。

### 规则 2：个性化规则（标签匹配）

```
你的兴趣画像：{"故事": 0.8, "情感": 0.6}

帖子A 标签：["故事", "治愈"]  → 匹配"故事"权重 0.8，高分
帖子B 标签：["科技", "编程"]  → 一个没匹配，0 分
```

### 规则 3：协同过滤（简化版）

记录你浏览最多的类别 → 同类别帖子自动加分。

```
你浏览了 25 次故事、12 次随笔 → 故事类帖子自动加权
```

### 规则 4：运营加权

管理员手动加精/置顶 +80~100 分，有效期 24 小时。

### 优化 1：多样性优化（核心亮点）

**问题**：同一位作者发了 5 篇热门帖子，不优化就会霸榜。

**解决**：每推荐一篇，就给"同类别"和"同作者"的后续文章扣分。

```
第 1 篇：张三《故事A》90 分 → 被选中
        记录：故事类×1，张三×1

第 2 篇：张三《故事B》88 分 → 扣分：
        - 故事类出现 1 次  → 扣 0.2×0.5×1 = 扣 0.1
        - 张三出现 1 次    → 扣 0.2×0.3×1 = 扣 0.06
        调整后：87.84 分

第 4 篇：张三《故事C》85 分 → 扣分：
        - 故事类出现 2 次  → 扣 0.2×0.5×2 = 扣 0.2
        - 张三出现 2 次    → 扣 0.2×0.3×2 = 扣 0.12
        调整后：84.68 分

张三的帖子越到后面扣越多，给不同类别的文章让路。
```

### 优化 2：得分明细（调试模式）

不做黑盒推荐。每条结果附带每条规则的贡献：

```javascript
{
  score: 78.5,
  scoreBreakdown: {
    "热度规则":   { rawScore: 25,  weight: 0.3,  weightedScore: 7.5 },
    "个性化规则": { rawScore: 60,  weight: 0.35, weightedScore: 21 },
    "协同过滤":   { rawScore: 40,  weight: 0.2,  weightedScore: 8 },
    "运营加权":   { rawScore: 280, weight: 0.15, weightedScore: 42 },
  }
}
```

产品经理问"为什么推这篇不推那篇"时能精确回答。

### 优化 3：配置导入导出

```javascript
engine.exportConfig()  // 导出规则权重 + 配置 → JSON
engine.importConfig()  // 从 JSON 恢复
```

A/B 测试时一键切换配置，不用改代码。

### 面试完整回答

> "推荐引擎用策略模式解耦四条打分规则。热度用互动加权×时间衰减，个性用标签匹配，协同用浏览历史，运营人工干预。三个关键优化：一是多样性优化，同类目或同作者重复出现时动态降权，防止同质化；二是得分明细，能看到每条规则的独立贡献；三是配置导入导出，支持 A/B 测试一键切换权重。"

## 17. 云函数：多 Action 设计模式

### 问题

每个操作建一个云函数 → 5 个云函数（新增评论、删除评论、点赞评论、举报评论、置顶评论），重复配置部署。

### 解决

一个云函数根据 `action` 参数分流：

```javascript
// commentAction/index.js — 一个函数管全部
exports.main = async (event) => {
    const { action, commentId, content } = event;
    
    switch (action) {
        case 'add':    // 新增评论 → 校验权限 → 写入 → 更新帖子评论数 → 通知作者
        case 'delete': // 删除 → 校验是不是自己的 → 删除 → 评论数-1
```

## 19. Spring IoC/DI 与 AOP

### IoC/DI：你别 new 了，Spring 帮你 new

**不用 Spring 时**：`private UserService userService = new UserService();` 自己 new，还要管它依赖什么。

**用 Spring 后**：

```java
@Autowired
private UserService userService;  // 没写 new，但 userService 不是 null
```

Spring 启动时扫描所有 `@Component/@Service/@Repository/@Controller` → 全部 new 好（叫 Bean）→ 放进一个大 HashMap → `@Autowired` 从 HashMap 里取出来注入。

**类比**：不用 Spring = 自己烧水倒水，用 Spring = 说"我要喝水"管家递过来（控制反转），管家把水递手里（依赖注入）。

**四个注解**：`@Component` `@Service` `@Repository` `@Controller` 功能完全一样（都让 Spring 管理），只是名字不同让人一眼看出是哪一层。

### AOP：不改原代码加功能

**场景**：100 个方法都要加日志，改 100 次？

**不改一行原代码**：

```java
@Aspect
@Component
public class LogAspect {
    @Around("execution(* com.xxx.service..*.*(..))")
    public Object log(ProceedingJoinPoint jp) throws Throwable {
        System.out.println("开始执行 " + jp.getSignature().getName());  // 前置
        Object result = jp.proceed();   // 执行原方法
        System.out.println("执行完毕");   // 后置
        return result;
    }
}
```

**你每天都在用的 AOP**：`@Transactional` 注解——Spring 自动帮你包了 `try { 开启事务 → 你的代码 → 提交 } catch { 回滚 }`。

### 面试回答

> "IoC 是控制反转，对象创建权交给 Spring 容器。DI 通过 @Autowired 注入依赖。AOP 是面向切面，在不改原代码的前提下加横切功能，典型场景是 @Transactional 事务管理。"

## 20. MySQL：索引与事务

### 索引

没索引 = 100 万行逐行扫，有索引 = 翻目录直达，100 万次变 ~20 次。底层是 B+ 树。

什么加索引：高频查的字段加。什么不加：区分度低的（如 status 只有 0/1）。

**LIKE '%三' 不走索引**：索引从左匹配，不知道前面是什么就没法定位。

### 事务

`@Transactional`：多条 SQL 绑在一起，要么全成功（commit），要么全作废（rollback）。

ACID：原子性（打包做）、一致性（别乱）、隔离性（别互相干扰）、持久性（别丢）。

MySQL 默认隔离级别"可重复读"。

## 21. Redis 缓存三大问题

缓存挡在 MySQL 前面。出问题 = 缓存没挡住，数据库被流量淹了。

| 问题 | 原因 | 修法 |
|------|------|------|
| **击穿** | 一个热点 key 过期，1 万请求砸同一个 SQL | 加锁，只放一个人查库 |
| **穿透** | 查不存在的数据，缓存永远挡不住 | 布隆过滤器 或 查不到也缓存空值 |
| **雪崩** | 大量 key 同时过期，所有请求同时砸 MySQL | 过期时间加随机值，别凑一块 |

## 24. Skill-First 架构（Tool → Skill → Orchestrator 三层）

### 三层各干什么

```
Tool            — 原子函数，只管执行，不做判断。如 search_vector()
Skill           — 有脑子的能力单元。知道什么时候调哪个 Tool，自带容错和校验
Orchestrator    — 老板。管意图识别→路由→调度，不亲自干活
```

### 为什么要分层

```
❌ 不分层：用户输入 → 巨大 if-else → 直接执行
   加新功能 = 改 if-else → 越改越乱

✅ 分层：加新功能 → 写一个 Skill（继承 BaseSkill）
   Orchestrator._auto_discover_and_register_skills() 自动发现
   不动 Orchestrator、不动其他 Skill
```

### 面试回答

> "Tool 是原子函数只管执行，Skill 封装判断逻辑和容错成独立能力单元，Orchestrator 管意图识别和调度。加新 Skill 只写一个类，编排器自动发现注册路由，零侵入。"

## 28. 三层容错架构（详解版）

### 第一层：Pydantic 门口拦

参数不对？门都不让进。"text 拼成 texxt"→ 当场 ValidationError。

### 第二层：指数退避重试

网络抖一下 → 等 1s 再试 → 还失败等 2s → 4s → 3 次放弃。不是马上重试——等一等给下游喘息时间。

### 第三层：熔断降级（门卫类比）

```
正常（CLOSED）   → 来人就开门
连偷 5 次        → 封门（OPEN），谁都不让进
冷却 30 秒       → 开条缝（HALF_OPEN），放一个人试试
  好人 → 恢复开门
  又是贼 → 立刻再封
```

**不是临时网络抖，是这个功能本身挂了——直接封禁，防止一个坏模块拖垮整个系统。**

### 面试回答

## 31. 对话记忆引擎（ConversationMemory）

### 问题

聊了 50 轮后，QueryRewrite 只取最近 3 轮历史——早期的上下文全丢了。

### 方案：摘要存向量，可检索

```
第 20 轮 → 调 LLM 生成摘要 → Embedding → 存 ChromaDB collection: "chat_summary_{id}"
第 40 轮 → 再触发 → 存第二条摘要

检索时：
  搜索知识库 + 搜索 chat_summary_{id}
  → 问题"上次那个报错" → 摘要匹配到"用户在排查 SSL 报错"
```

### 关键设计

- **异步执行**：`threading.Thread(target=_do_summary, daemon=True)`——不卡当前用户
- **保留最新 5 条**：旧摘要自动删除，防止存储膨胀
- **权重控制**：知识库 70%，摘要 30%

### 旧摘要删除后信息丢了怎么办

摘要不是全部历史——最新摘要是基于最近 30 轮生成的，已覆盖当前状态。优化方向：递进式摘要（新摘要融合旧摘要）。

### 面试回答

> "长对话摘要存在独立向量集合，检索时多搜一个 collection。异步执行不卡主流程，保留最新 5 条。本质是让历史对话也能被语义检索到，弥补早期上下文丢失。"

## 39. LLM 意图路由（取代关键词匹配）

### 为什么不用关键词

"帮我看看这段代码，然后用中文总结一下"——关键词匹配到"代码"和"总结"两个，该走解释代码还是摘要？多意图无法处理。

### LLM 方案

```python
# 一次 qwen-turbo 调用同时返回两项判断
def _classify_intent_with_llm(user_input):
    prompt = f"""输出 JSON：
{{"pipeline": "流水线名或null",
  "rag_mode": "vector 或 graphrag",
  "intent": "简短描述"}}"""
    response = llm.chat(messages=[{"role":"user","content":prompt}], temperature=0.0)
    return json.loads(response.content)
```

一次调用同时知道：走哪条流水线 + 该用哪种 RAG 模式。比原来的两次调用少花一半时间。

### 生产缓存

高频意图可本地缓存——80% 命中缓存直接执行（零延迟），20% 未命中才调 LLM。

### 面试回答

> "从规则路由升级到模型路由。qwen-turbo 一次调用同时返回流水线分类 + RAG 检索模式，取代了原有 MACRO_KEYWORDS 词表匹配和关键词流水线匹配。多意图场景（'看看代码再中文总结'）关键词无法处理，LLM 能准确解析。"

## 40. Rerank 为什么不能省

向量检索按余弦相似度排序——"Python" 能匹配到《Monty Python 喜剧团介绍》（向量相似但内容无关）。

gte-rerank 逐对阅读"问题↔文档"内容做深度判断——喜剧团直接被打到 0.12 分。

**向量检索 = 粗筛（快），Rerank = 精排（准）。30 篇 → Rerank → 取 5 篇。**

## 41. 为什么用 SQLite 而不是 MySQL

**① 零部署**：一个文件 `checkpoints.sqlite`，`git clone` 就能跑。
**② LangGraph 原生支持**：`SqliteSaver` 官方自带，换 MySQL 得自己写几百行适配。
**③ WAL 模式**：并发读没问题，写锁是弱点。

生产环境：SQLAlchemy 抽象层，换 PostgreSQL 只改连接串，代码不变。

## 44. 分块策略详解：固定窗口 vs 语义段落

### 两种策略在你的代码里的实现

**固定窗口（fixed）**——`_chunk_by_chars`：直接按字符位置硬切，`chunk_size=500`，到 500 字就一刀。

**语义段落（recursive）**——推荐策略：先按 `\n\n` 段落切 → 段落太大按 `\n` 句切 → 还太大按 `。` 切 → 最后才按字符切。**语义边界优先，字符限制兜底。**

### 面试官会问的核心例子

2000 字技术文档（try/except → 自定义异常 → finally → 最佳实践）：

**固定窗口 500 字硬切**：
- Chunk 1 [450-950]：从"自定义异常类"段落中间劈开——只含"自定"两个字
- Chunk 2 [900-1400]：只含"义异常类"后半部分
- 用户问"什么时候用自定义异常类" → 两个残缺块 embedding 都匹配不上 → 召回隔壁段落 → **答非所问**

**语义段落按空行切**：Chunk 2 开头就是"当内置异常不足以表达特定业务错误时" → 和问题天然匹配 → **Top-1 直接命中**

### 差距量化

| 维度 | 固定窗口 | 语义段落 |
|------|---------|---------|
| 语义完整性 | ❌ 50%+ 块在中间切开 | ✅ 96%+ 保持语义完整 |
| Top-K 召回 | K=5 才找到相关片段 | K=1~2 就够 |
| LLM 幻觉风险 | 拿半句话，LLM 靠猜 | 拿完整段落，直接引用 |

### 你比"段落分块"多了三件事

**① 递归降级**：先用 `\n\n` 切 → 太大用 `\n` → 还大用 `。` → 最后字符兜底。
**② 语义小块合并**：＜20 字的碎片合并到语义相关邻居，不独立存垃圾块。
**③ 上下文窗口**：每个 chunk 携带上块末句 + 下块首句。

### 面试回答

> "固定窗口最典型失败场景：'自定义异常类'段落跨在两块边界，两个残缺块的 embedding 都匹配不上。recursive 分块语义边界优先、字符限制兜底——96% 的块是完整语义单元。还做了递归降级、小块合并、上下文窗口三个额外保护。"

## 45. 用户代理（User Proxy）与权限控制

### 你的项目当前模式：Auto Mode

`run_agent` 逻辑是 LLM 决定 → 立刻执行，中间没有用户确认步骤。


### Auto vs Human-in-the-Loop

| 模式 | 优点 | 缺点 |
|------|------|------|
| 全自动（你的） | 效率高，一句话完成 | 用户无控制权，错了才知道 |

| **混合模式（推荐）** | 正常批确认，异常精确问 | 兼顾体验和安全 |

### 业界方案：批处理确认 + 异常精确询问

```
LLM 拆解计划 → 用户一次性确认 → 全自动执行
如果不成功 → 停下来问用户
```

### 权限控制三层（你项目里目前没有）

```python
class TranslatorSkill(BaseSkill):

class FileDeleteSkill(BaseSkill):
    permission_level = "high_risk"     # 必须用户确认
```

**第二层：资源级权限**——路径白名单/黑名单，不在白名单里则询问用户。

**第三层：审计日志**——每次工具调用记录 user_id、action、参数、结果。

### 面试回答

## 46. SSE 长节点沉默与多用户隔离

### 问题 1：长节点沉默（8 秒没消息）

你的状态推送只在 `on_chain_start` 节点开始时触发。如果一个节点（如 `vector_search`）执行了 8 秒——**用户 8 秒收不到任何事件**，前端以为卡死了。

**方案一**：拆长节点为多个小节点（如 `search_start → search_progress → search_done`），每个 <2s，`on_chain_start` 自然产生推送。

**方案二**：SSE 心跳协程，每 5 秒推一次 `heartbeat`，告知前端连接还在。

### 问题 2：多用户会串流吗？

**主 SSE 流安全**——每个 HTTP 请求创建独立的 AsyncGenerator，用户 A 和用户 B 的事件完全隔离，每条事件还带 `request_id` 双重校验。

**风险在异步摘要线程**——`_trigger_summary_async` 用了全局单例 `get_conversation_memory()`，锁粒度为整个 memory 对象。A 和 B 的摘要线程会互相阻塞（性能问题，不是数据串流问题）。

### 面试回答

> "主 SSE 流安全隔离——每个请求独立 AsyncGenerator，数据不会串流。但异步摘要线程的全局锁会导致多用户摘要生成串行化，这是可优化点。长节点缺心跳机制，优化方向是加 SSE 心跳或拆长节点为多个小节点。"


|------|-------------|----------|

### 不存在优先级问题——入口决定路径

- API 路由 `/chat` → 永远走 LangGraph
- `process()` → 永远走 Pipeline/单技能

## 48. 规则引擎 vs 查询改写（职责分工）

### 各管各的事

| 维度 | 规则引擎（inject_rules） | 查询改写（rewrite_query） |
|------|------------------------|--------------------------|

## 49. LangGraph Checkpointer 详解

### 后端：SQLite

```python
from langgraph.checkpoint.sqlite import SqliteSaver
checkpointer = SqliteSaver.from_conn_string("checkpoints.sqlite")
graph.compile(checkpointer=checkpointer)
```

数据存 `checkpoints.sqlite` 文件。**服务崩溃后重新打开，状态全都在。**

### 多轮对话靠 thread_id

```python
# 创建 session 时生成一次 UUID，此后不变
langgraph_thread_id = str(uuid.uuid4())
config = {"configurable": {"thread_id": langgraph_thread_id}}
```

### 三问三答时间线

```
Q1: invoke({current_input: "..."}, {thread_id: "550e"})
    → 首次无历史 → 全流程执行 → 每节点写 checkpoint → state 持久化

Q2: invoke({current_input: "那连接池大小..."}, {thread_id: "550e"})
    → 自动加载 Q1 checkpoint！state = {messages: [q1, a1], ...}
    → merge 新的 current_input → rewrite_query 看到 [q1, a1] 展开"那"

Q3: invoke({current_input: "这个配置..."}, {thread_id: "550e"})
    → 加载 Q2 checkpoint → messages = [q1, a1, q2, a2]
    → "这个配置"→"数据库连接池配置的安全风险"
```

### 持久化了哪些 state 字段

| 字段 | 作用 |
|------|------|
| `messages` | 对话历史（`add_messages` 追加不覆盖） |
| `current_output` | 上一轮答案 |
| `rewritten_query` | 改写后查询 |
| `search_results` / `reranked_results` | 检索/精排结果 |
| `cumulative_token_usage` | Token 累计 |

### 面试回答

> "Checkpointer 用 SQLite 后端存 `checkpoints.sqlite`，崩溃后恢复。多轮对话靠 thread_id——创建 session 时生成，之后所有请求都用同一个。Q2 调用时 LangGraph 自动从 SQLite 恢复 Q1 的完整 state（含 messages 和检索结果），和新输入合并继续执行，对话上下文自然保持。"

## 50. 测试策略：SimulatedLLM + 三层隔离

### LLM 输出不确定怎么写 assert？

**SimulatedLLM**——纯关键词匹配模拟器：从技能工具的 name/description 提取关键词和用户输入匹配，返回确定的 tool_calls。**同样输入 + 同样工具列表 = 同样结果。**

**Assert 策略**：不校验精确文本，校验：
```python
assert result.success                    # 流程正确性
assert result.pipeline_type == "single"  # 路由类型
assert "Guido" in answer                 # 关键信息包含（不逐字）
```

### 三层依赖隔离

**第一层：Mock**——`patch` 掉网络调用（vector_search、rerank），放行真实逻辑（rag_answer 走 SimulatedLLM）。
**第二层：内存数据库**——`get_checkpointer(":memory:")`，测试间自动隔离。
**第三层：条件跳过**——`@pytest.mark.real` 标记，无 API Key 自动跳过。

```bash
pytest tests/ -v -m "not real"   # 日常：仅单元测试
pytest tests/ -v                  # 回归：全量（需 Key）
```

### 面试回答

> "单元测试占 80%，SimulatedLLM 确定性地模拟 LLM 返回，不依赖网络和 Key。集成测试宽松 assert 关键实体 in answer，模型升级也不挂。三层隔离：Mock 网络调用、SQLite :memory: 内存库、条件 skip 跳过真实测试。"

## 50-附2. 测试有失败的怎么解释

### 面试可能问

> A：有，但不多。上次跑了 1182 个，过了 1160 个，挂的 22 个分两类：一类是 LiteLLM 库版本升级后构造函数签名变了，4 个 mock 没跟上（修起来很快）；另一类是 GraphRAG 和 Reflection 的集成测试，依赖本地图数据库服务没启动，在 CI 环境里会自动 skip。核心单元测试都是全过的。

### 确实有两份历史

| | Session Manager `message_history` | Checkpointer `messages` |
|--|----------------------------------|--------------------------|
| 格式 | JSON（Message 对象） | msgpack（BaseMessage） |
| 用途 | 前端展示、历史查询 | LangGraph 状态恢复 |
| 谁来读 | 前端 `/history` API | LangGraph 内部 |

**设计上故意存在**——Session Manager 是展示层，Checkpointer 是执行层。

### 删一条消息怎么办？

当前没有单条删除 API。如果要实现：
1. 从 `message_history` 删除
2. 在 metadata 记录"已删除消息 ID"
3. `query_rewrite` 过滤掉已删消息
4. Checkpointer 不动——新 checkpoint 覆盖旧的

### 多用户鉴权：三层依赖链

```python
# ① 从 API Key 解析 user_id
verify_api_key(key) → user_id

# ② 查 session + 校验所有权
get_owned_session(session_id, user_id):
    session = sm.get_session(session_id)
    if session.user_id != user_id → HTTP 403

# ③ 路由处理器看到的已是鉴权后的 session
@router.get("/{session_id}")
def get_session(session = Depends(get_owned_session)):
    # 403 时代码根本不执行
```

A 伪造 B 的 session_id → `user_A != user_B` → 精确 403 拦截。

API Key 以 sha256 哈希存，开发环境 `env=="dev"` 自动免鉴权。

### 面试回答

> "两份历史设计上各有用处：Session Manager 展示层（JSON 给前端），Checkpointer 执行层（msgpack 给 LangGraph）。多用户鉴权三层依赖注入：verify_api_key 从 Key 哈希解 user_id → get_owned_session 对比 session.user_id → 不匹配 403。A 伪造 B 的 session_id 会被精确拦截。"

## 52. LiteLLM 的 5 层封装

### LiteLLM 解决什么

没有 LiteLLM：对接 3 个模型写 3 套代码（OpenAI SDK、DeepSeek 自己设 base_url、豆包用 volcengine SDK）。
有 LiteLLM：`litellm.completion(model="deepseek/deepseek-chat", ...)` 一个函数覆盖 100+。

### 你在它上面加的 5 层封装

**① Pydantic 类型安全**
```python
# LiteLLM 返回裸 dict → 你包装成
class ChatResponse(BaseModel):
    content: str
    tool_calls: list[ToolCall]
    usage: dict
```

**② 指数退避重试**——只重试限流/服务不可用（3 次，2s→4s→8s），密钥错误直接抛。

**③ 结构化日志 + metrics 埋点**——每次调用记录 trace_id、模型名、耗时、token。

**④ tool_calls 统一解析**——兼容参数是 string 还是 dict，解析失败也不崩（兜底 `{"_raw": args}`）。

**⑤ ToolChoice 逻辑处理**——支持指定调用某个具体工具（不限于 auto/none/required）。

### 切模型改什么

```bash
# .env 改 4 行，代码不动
LLM_PROVIDER=deepseek    # 或 dashscope/doubao
LLM_MODEL=deepseek-chat  # 或 qwen-plus
LLM_API_KEY=sk-xxx
```

LiteLLM 根据 `provider/` 前缀自动路由 API 地址和参数格式。**边界情况**：不支持 Function Calling 的模型会直接抛异常——缺少降级逻辑。

### 面试回答

"LiteLLM 解决多提供商适配。我加了 5 层：Pydantic 类型安全、指数退避重试、日志 metrics 埋点、tool_calls 统一解析、ToolChoice 处理。切模型只改 .env 四行代码不动。"

## 53. ID 三套方案（心笔小程序——踩过最大的坑）

### 三套 ID 是什么

| 字段 | 类型 | 用途 | 示例 |
|------|------|------|------|
| `_id` | ObjectId | 数据库内部管理 | 不可预知 16 进制 ID |
| `_openid` | String | 微信权限验证 | oABC123xyz |
| `id` | String | 业务查询和关联 | INK_001 |

### 为什么会踩坑

7 篇技术文档讲 ID 问题（`【10】ID字段改造` ~ `【21】用户主页ID字段修复`）。

**坑 1：跨集合关联时不知道用哪个 ID**

```javascript
// 关注功能——要兼容三种 ID 格式！
let targetUser = await db.collection('users').where(
  _.or([{ _id: targetUserId }, { _openid: targetUserId }, { id: targetUserId }])
).get()
```

**坑 2：代码写 `_openid`，索引建在 `userId` 上——查询不命中索引**

```javascript
// postAction 查点赞记录
const exist = await db.collection('likes').where({
  _openid: wxContext.OPENID,   // ← 用 _openid
  targetType: targetType,
  targetId: postId
}).get()
// 但 likes 表索引配在 (userId, targetType, targetId) 上 → 索引失效
```

### 最终规范

```
✅ 正确做法：
  _openid     → 仅用于微信权限认证（安全规则）
  _id         → 仅用于单文档操作（doc(id).get()）
  id（业务ID）→ 跨集合关联统一用这个

❌ 错误做法：
  不要用 _openid 做业务查询
  不要用 _id 做跨集合关联
```

### 面试回答

> "心笔有三套 ID——`_id` 数据库内部用、`_openid` 微信权限用、`id` 业务关联用。踩过的坑：关注功能要兼容三种 ID 格式查询、点赞表代码写 _openid 但索引建在 userId 上导致索引失效。最终规范：_openid 只用于权限验证，_id 只用于单文档操作，业务关联统一走自定义的 id 字段。"

## 54. 微信云开发架构局限性与应对

### 三大局限

**① 云函数冷启动**——首次调用或长时间未调用后容器初始化耗时 1~3 秒。

应对：
```javascript
onLoad() {
  wx.cloud.callFunction({ name: 'warmup' })   // 提前激活容器
  this.loadPosts()
}
```

**② 执行时长限制（默认 6 秒）**——推荐引擎用 `Promise.all` 并行召回应对，但协同过滤串行遍历容易超时。

**③ 数据库调用次数限制**——一个点赞操作做了 6 次 DB 调用。1000 用户一天点 50 赞 = 30 万次，免费额度用完。

优化：`Promise.all` 合并并行查询，减少调用次数。

### 面试回答

> "微信云开发三大局限：冷启动 1~3 秒、单次执行 6 秒上限、数据库调用次数有限制。应对：预热激活容器、Promise.all 并行调用、合并查询减少次数。企业级应用要考虑迁移到 TCB 或自建后端。"

## 55. 前端用户体验优化与成长体系（心笔小程序）

### 瀑布流无限滚动

```javascript
data: { pageSize: 4 }   // 每次 4 条，体感快不卡顿
```

需加节流：
```javascript
onReachBottom() {
  if (this.data.loadingMore || this.data.noMore) return
  this.setData({ loadingMore: true })
  this.loadMorePosts()
}
```

### 乐观更新（点赞）

```javascript
async handleLike() {
  const wasLiked = this.data.liked
  this.setData({ liked: !wasLiked })    // ① 立即改界面（零延迟）
  try {
    await api.post.like(this.data.postId) // ② 异步调接口
  } catch {
    this.setData({ liked: wasLiked })     // ③ 失败回滚
  }
}
```

### 用户成长体系

```javascript
{ id: "INK_001", level: 7, title: "七阶文豪", power: 12840, badges: [...], postsCount: 45, likesCount: 523000 }
```

**遗留问题：数据一致性**——发帖/点赞时不同步更新 `users` 表统计字段，导致显示可能不一致。

改进方向：云函数里同步更新（`Promise.all` 同时更新 posts 和 users）或定时任务批量同步。

### 面试回答

> "瀑布流每次 4 条加载体感快；乐观更新先改界面再调接口、失败回滚；骨架屏减少白屏。成长体系 level/title 存数据库，power 按公式估算，但存在 counts 字段不同步的遗留问题——改进方向是云函数里同步更新或定时任务批量同步。"
